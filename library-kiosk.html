MSU Library Interactive Kiosk

Redesigning a public touchscreen wayfinding system for physical interaction constraints and multi modal accessibility.

Role: UX Researcher
Timeline: 4 weeks
Team: 3 researchers

Redesigned idle state with clear affordance and persistent accessibility icon

The Outcome

I redesigned an interactive library kiosk with a 0% wayfinding success rate. Through usability testing and competitive analysis, I identified critical failures in touchscreen affordance, physical interaction, and accessibility—challenges applicable across automotive, medical, and public touchscreen systems.

The core problem: Students couldn't navigate to rooms because the display looked static (poor affordance), touch sensitivity required multiple taps, and zero accessibility features existed despite ADA requirements for public facilities.

Key Findings

0% Task Success Rate

None of 3 participants could locate their target room. Maps showed layouts but provided no "you are here" indicator or directional guidance.

2-3× Taps Required

Touch sensitivity issues required 67-100% more taps than industry standard (target: <1.2 taps per interaction).

0/7 WCAG Standards Met

Zero accessibility features implemented. No contrast adjustment, text sizing, audio support, or keyboard navigation.

Research & Methods

Moderated Usability Testing

Recruited 3 participants representing diverse library users. Used think-aloud protocol with 3 standardized tasks:

  • Find the next MSU event
  • Find a librarian's email address
  • Navigate to the Red Cedar Instruction Room

Competitive Analysis

Benchmarked against Broad College of Business and Law Building displays. Broad's system achieved significantly higher task completion through visual wayfinding (floor maps + "you are here" + turn-by-turn directions).

Heuristic Evaluation

Systematic review against Nielsen's usability principles and WCAG 2.1 AA accessibility standards. Identified 7 critical violations affecting discoverability, feedback, and universal access.

The Critical Insight

"I had no idea the screen was interactive—I thought it was just displaying library hours."

— Ashley, 22, Kinesiology major

Users walked past without realizing they could interact. The fundamental problem: poor affordance. The display looked static—no visual cues, no motion, no prompts, no clear call-to-action.

Second critical finding: Even when users did interact, maps showed room locations but provided no guidance on how to get there. All 3 participants expressed confusion about spatial terms like "West Wing, 2nd Floor" and couldn't translate map positions into physical navigation.

"I guess I'm supposed to just know it's on the West wing second floor? I don't believe I could actually find the room."

— Ashley

The Solution

1Fixed Affordance Problem

Redesigned idle state with prominent "TOUCH SCREEN TO BEGIN" prompt using high-contrast typography and MSU branding. Made the accessibility icon permanently visible in the top-right corner.

Design rationale: Static text outperforms motion graphics for accessibility—motion can trigger vestibular disorders and would be distracting in quiet library environment.

Similar patterns in: Automotive "touch to wake" displays (Mercedes MBUX, BMW iDrive), medical device standby modes, industrial control panels. All balance visibility with distraction prevention.

2Built Multimodal Accessibility System

Added 7 configurable accessibility options based on WCAG 2.1 AA requirements:

  • Contrast: High contrast mode for low vision users
  • Plain English: Simplified language for cognitive accessibility
  • Keyboard-only: Navigate without touch (motor disabilities)
  • Logic & Headers: Semantic structure for screen readers
  • LARGE: Increased font size (18pt → 24pt)
  • Image + alt: Alt text for all visual elements
  • Fonts: Accessible typefaces (OpenDyslexic option)

Plus icon-based navigation preferences for audio, cognitive, and sound-based wayfinding.

Why these 7? Based on MSU disability services data showing most-requested accommodations. Each toggle enables/disables specific WCAG success criteria.

Broader application: Automotive displays increasingly offer similar customization (Honda Accord's configurable contrast, Tesla's accessibility modes). Medical devices require multimodal input for diverse patient needs. Industrial systems need glove-friendly interfaces with audio feedback.

3Implemented Progressive Disclosure Wayfinding

Inspired by Broad College's successful system. Four-step progression:

  • Show floor overview map first (spatial context)
  • Highlight destination room on map (visual confirmation)
  • Display "YOU ARE HERE" indicator with directional arrows
  • Provide turn-by-turn text instructions

Design principle: Users need both spatial context (map view) and step-by-step guidance (directions). Mirrors modern automotive navigation UX—early GPS showed only maps; modern systems combine map + turn-by-turn + voice + lane guidance.

4Simplified Navigation Architecture

Consolidated duplicate features (Maps vs. Find a Location vs. Directions) into task-based categories: Conference Rooms, Study Rooms, Restrooms, Elevators, Departments.

Added search/filter: "Filter by Name, Floor, Room #, Type of Classroom"

Why task-based? Users think in goals ("find a study room"), not system features ("check Maps tab vs. Find a Location tab"). Reduces cognitive load and navigation time.

What I Got Wrong

We Never Validated the Redesign

⚠ The critical gap: We designed solutions based on 3 user tests but never tested whether our fixes actually work.

What we should have done:

  • Tested redesigned prototype with 10+ new participants
  • Measured whether task success improved from 0% to target 75%
  • Validated accessibility features with users who need them (partner with disability services)
  • A/B tested on physical hardware to confirm touch sensitivity improvements

Why we didn't: 4-week academic project timeline

The learning: In professional HMI work, validation isn't optional—it's the primary deliverable. Automotive manufacturers, medical device companies, and industrial control designers don't ship based on assumption. They require empirical evidence through user clinics, simulator testing, and field validation.

Sample Size Too Small

3 participants ≠ generalizable findings

Industry standard: 5 for qualitative insights, 15-30 for quantitative task analysis. Our findings identify problems but can't statistically validate solutions.

If I did this again:

  • Recruit 15+ participants across diverse groups
  • Use statistical analysis to validate patterns
  • Test edge cases (winter gloves, wet hands, wheelchair users at different heights)

Missing Quantitative Rigor

What we captured: Qualitative observations ("Ashley struggled," "took a while")

What HMI roles require: Precise metrics (task time: 2m 14s ± 18s, error count: 4.2 per task, SUS score: 42/100)

We noted "about 2 minutes" instead of timing precisely. We observed "multiple taps" instead of counting exact errors. For professional HMI work, these become engineering specifications, not just observations.

If This Project Continued

Three-phase validation plan:

Phase 1 - Week 1

Validate Redesign

Test Figma prototype with 10 participants. Measure task success rate, completion time, SUS score.

Target: Task 3 success ≥75%, time <90s, SUS score ≥70

Phase 2 - Weeks 2-3

Hardware Testing

Deploy on actual kiosk. A/B test old vs. new interface with 100+ real interactions.

Target: Engagement rate ≥25% (5× improvement), touch sensitivity <1.2 taps

Phase 3 - Week 4

Accessibility Validation

Partner with disability services. Test screen reader, keyboard navigation, high contrast with users who rely on these features.

Target: 100% WCAG 2.1 AA compliance verified with assistive technology
